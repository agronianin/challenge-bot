# Promt

## Назначение файла

Этот файл является инструкцией по проекту.
Если полностью потеряется контекст разработки, новый разработчик должен прочитать:

1. этот файл;
2. `Status.md`;
3. `README.md`.

После этого он должен понимать:

- что за продукт разрабатывается;
- какие требования уже зафиксированы;
- какие архитектурные решения уже приняты;
- что в проекте уже реализовано;
- что еще не реализовано;
- в каком порядке безопасно продолжать работу.

## Что за проект

Разрабатывается Telegram-бот для групповых fitness challenge в чатах.

Основная идея:

- администратор запускает challenge в групповом чате;
- бот ежедневно публикует задания с упражнениями;
- участники присоединяются к challenge;
- участники получают личные задания и напоминания;
- участники отмечают выполнение;
- бот ведет статистику по прогрессу.

## Цель MVP

Минимально полезная версия должна уметь:

- запускаться на Ubuntu-сервере в Docker;
- подключаться к PostgreSQL;
- подключаться к Telegram;
- хранить упражнения в БД;
- запускать challenge из группового чата;
- публиковать хотя бы один день challenge;
- позволять пользователю присоединиться;
- позволять пользователю отметить выполнение дня;
- сохранять историю выполнения;
- отправлять личные напоминания;
- переживать рестарт контейнеров без потери данных.

## Принятые технические решения

### Конфигурация и секреты

- Реальные секреты не хранятся в репозитории.
- Шаблон конфигурации хранится в `.env.example`.
- Реальные значения хранятся в локальном `.env`.
- На сервере также используется отдельный локальный `.env`.
- `.env` игнорируется через `.gitignore`.

### Миграции базы данных

- Liquibase не должен запускаться автоматически при старте приложения.
- Миграции применяются отдельной явной командой.
- Разработчик должен видеть результат применения миграций в терминале.
- Источник миграций: `db/liquibase/changelog.xml`.

### Деплой

- PostgreSQL и бот запускаются в Docker Compose.
- Миграции запускаются отдельным compose-сервисом `migrate`.
- Обновление приложения и обновление PostgreSQL считаются разными операциями.
- PostgreSQL не нужно перезапускать при каждом обновлении приложения.

### Логирование

- Логи приложения должны писаться в доступный путь.
- Используется `logback-spring.xml`.
- По умолчанию используется каталог `./logs` локально и bind-mounted каталог в Docker.
- Должно быть два файла логов:
  - `application-info.log` для логов уровня `INFO` и выше
  - `application-debug.log` для более подробных логов приложения
- Оба файла должны ротироваться.
- Debug и trace логи сторонних библиотек не должны попадать в файлы без явного включения.
- SQL-запросы Hibernate и bind-параметры должны попадать в `application-debug.log`.

### Telegram startup

- Если `BOT_TOKEN` пустой или равен `replace_me`, приложение может стартовать без polling.
- Это сделано для безопасной проверки приложения без реального токена.

## Продуктовые требования

### Роли

Есть три роли:

- администратор чата;
- участник challenge;
- система.

### Что делает администратор

Администратор должен иметь возможность:

- открыть административную панель в личном диалоге с ботом;
- добавить другого администратора;
- добавить бота в групповой чат;
- запустить challenge;
- задать параметры challenge;
- при необходимости остановить или завершить challenge;
- загружать упражнения;
- смотреть статистику.

Только администратор должен иметь доступ к административным командам.

### Как появляется первый администратор

На текущем этапе первый администратор заводится вручную в базе данных.

Текущее правило:

- в таблице `users` должна существовать запись с нужным `tg_id`;
- у этой записи поле `role` должно быть равно `admin`;
- после этого пользователь может открыть личный чат с ботом и получить административное меню.

Как practically получить запись пользователя в БД:

- любой пользователь должен написать боту любое сообщение или отправить `/start`;
- после этого бот должен создать или обновить запись пользователя в таблице `users`.

Что происходит после ручной смены роли в БД:

- если пользователю в базе выставить `role = admin`, удалять диалог с ботом не требуется;
- достаточно отправить боту следующее сообщение или снова написать `/start`;
- после этого бот должен показать административное меню.

Это временное решение для MVP-этапа. Позже можно добавить отдельный bootstrap-механизм.

### Административная панель

На текущем этапе административная панель работает в личном чате с ботом.

Если пользователь является администратором и отправляет `/start`, бот должен показать удобное меню.

В меню должны быть доступны действия:

- `Добавить администратора`
- `Начать challenge`

### Добавление администратора

Текущее решение:

- администратор выбирает пункт `Добавить администратора`;
- бот предлагает выбрать пользователя через Telegram `requestUsers`;
- после выбора пользователя бот создает или обновляет запись в `users`;
- выбранному пользователю выставляется `role = admin`.

Важно:

- сейчас используется не передача контакта, а встроенный механизм выбора пользователя через Telegram-кнопку;
- это удобнее и надежнее для текущей реализации.

### Создание challenge через личный диалог

Текущее решение для MVP:

- администратор выбирает пункт `Начать challenge`;
- бот предлагает выбрать чат через Telegram `requestChat`;
- после выбора чата бот пошагово запрашивает параметры.

На текущем этапе пошагово вводятся:

- дата старта;
- длительность в днях;
- время публикации;
- timezone;
- количество упражнений в день;
- количество групп мышц в день.

После ввода параметров бот:

- показывает сводку;
- ждет подтверждение `Создать challenge`;
- сохраняет `Program` в БД;
- заранее создает `ProgramDay` на все дни challenge;
- отправляет подтверждение в выбранный чат;
- возвращает администратора в главное меню.

### Отмена действия

Во время любого административного сценария должна быть доступна кнопка `Отмена`.

Поведение:

- текущий сценарий прерывается;
- состояние диалога очищается;
- администратор возвращается в основное меню.

### Что делает участник

Участник должен иметь возможность:

- вступить в challenge;
- заполнить личные данные;
- получать ежедневные задания;
- получать личные напоминания;
- отметить выполнение дня;
- смотреть собственный прогресс.

## Упражнения

Упражнение должно содержать:

- `id`
- название
- группу мышц, при этом группа может быть опциональной
- путь к видеофайлу или Telegram `file_id`
- базовое количество повторений для коэффициентной части
- коэффициент упражнения
- флаг, что упражнение имеет полностью статичное количество
- статичное количество повторений
- тип коэффициента:
  - количество
  - секунды
- комментарий

Важные правила:

- если видео есть, бот должен уметь отправить его;
- если видео нет, бот должен отправить текст;
- упражнения должны храниться в БД;
- позже будет нужен импорт упражнений из CSV.
- упражнение может иметь фиксированную добавку `static_reps`, которая не зависит от коэффициентов;
- если `is_static_reps = true`, итоговое значение упражнения равно `static_reps`, а коэффициентная часть игнорируется;
- если `is_static_reps = false`, итоговое значение упражнения считается как сумма:
  - `static_reps`
  - значения, рассчитанного по коэффициентам.

## Группы мышц

Система хранит справочник групп мышц.

Примеры:

- ноги
- руки
- пресс
- спина
- грудь
- плечи
- full body
- без группы

Важно:

- в требованиях уже зафиксировано, что группа может быть опциональной;
- это нужно учесть при дальнейшей переработке схемы БД, потому что текущая схема еще не соответствует этому требованию.

## Challenge

Challenge запускается в конкретном групповом Telegram-чате.

При создании challenge администратор задает:

- чат;
- дату старта;
- длительность в днях;
- время ежедневной публикации;
- количество упражнений в день;
- количество групп мышц в день;
- правило роста нагрузки по дням;
- частоту выходных дней.

Ожидаемый сценарий:

1. Администратор вызывает команду.
2. Бот проверяет, что это администратор.
3. Бот создает challenge.
4. Бот создает связанные дни challenge.
5. Бот подтверждает параметры запуска.

На текущем этапе это уже реализуется через личный пошаговый сценарий в административной панели.

## Ежедневная публикация дня

Каждый день бот должен публиковать новый день challenge в групповой чат.

В групповом сообщении должно быть:

- номер дня challenge;
- список упражнений;
- комментарии;
- видео, если доступны;
- кнопка `готово`.

После публикации:

- данные дня должны сохраняться в БД;
- повторная публикация того же дня не должна создавать дубль;
- уже опубликованный день не должен пересобираться задним числом.

## Подбор упражнений

Алгоритм подбора на MVP должен учитывать:

- `exercises_per_day`;
- лимит по числу групп мышц;
- недопустимость повторов, если еще есть уникальные упражнения;
- нежелательность использовать группы мышц, которые были в предыдущий день.

Для MVP достаточно простого алгоритма:

1. взять допустимые упражнения;
2. убрать неподходящие группы и упражнения;
3. выбрать нужное количество;
4. сохранить результат.

## Персональная нагрузка

Итоговая нагрузка рассчитывается по формуле:

`result = exercise_multiplier * day_multiplier * user_multiplier`

С учетом нового статичного компонента итоговое правило должно быть таким:

- сначала считается коэффициентная часть;
- затем к ней прибавляется `static_reps`;
- если у упражнения `is_static_reps = true`, то итоговое значение равно только `static_reps`.

Требования:

- расчет детерминированный;
- округление всегда вверх;
- рассчитанное значение сохраняется в БД;
- так как значения могут отличаться у разных пользователей, персональные цифры нельзя полагаться выводить только в общем сообщении.

Из этого следует архитектурное правило:

- в групповой чат идет общий пост дня;
- пользователю в личку отправляется персональная версия задания.

## Участие пользователя

После публикации стартового поста challenge пользователь должен иметь возможность вступить в challenge.

При вступлении:

1. создается или активируется запись участия;
2. пользователю могут быть показаны его личные характеристики для редактирования;
3. пользователь становится участником challenge;
4. ему становится доступна отметка выполнения;
5. ему отправляется личное сообщение с текущим заданием.

Нужно хранить:

- участие пользователя в конкретном challenge;
- дату вступления;
- активность участия;
- персональный коэффициент;
- настройки напоминаний;
- флаг возможности писать в личку.

## Напоминания

Пользователь должен иметь возможность настроить личные напоминания.

Правила:

- время задает сам пользователь;
- напоминание отправляется активным участникам;
- если текущий день выполнен, напоминание не отправляется;
- если пользователь отстает, можно напоминать о последнем невыполненном дне;
- если бот не может написать первым в личку, это должно корректно обрабатываться.

## Отметка выполнения

Участник должен иметь возможность отметить выполнение дня.

Правила:

- отметка привязана к конкретному `program_day`;
- один пользователь может отметить день не более одного раза;
- время отметки сохраняется;
- должно быть различимо:
  - не участвует;
  - участвует, но не выполнил;
  - выполнил.

Отмена отметки допустима как будущее улучшение, но не обязательна для MVP.

## Надежность и устойчивость

Система должна быть устойчивой к повторным действиям и рестартам.

Ключевые требования:

- повторный запуск daily job не создает дублей;
- повторное нажатие кнопки участия безопасно;
- повторная отметка выполнения безопасно обрабатывается;
- история опубликованных дней не меняется;
- после рестарта контейнера работа продолжается за счет БД.

## Критические ограничения текущей кодовой базы

- Текущая схема БД еще не покрывает `program_participants` и `participant_reminders`.
- Текущая схема `exercises` еще не приведена к опциональной группе и новому типу коэффициента.
- `UpdateHandler` пока тестовый.
- `CsvImporter`, `ExerciseSelector`, `DailyRunner` пока не реализованы.
- первый администратор пока добавляется вручную в БД.
- Проверка реального Docker startup из текущего агентского окружения пока не выполнена, потому что здесь нет доступного Docker daemon.

## Как продолжать работу после потери контекста

1. Сначала прочитать `Status.md`.
2. Посмотреть, какой этап активен сейчас.
3. Проверить `README.md`, чтобы не сломать текущий deployment flow.
4. Перед изменением поведения обновлять этот файл.
5. Перед закрытием этапа обновлять `Status.md`.

Главное правило:

- `Promt.md` отвечает на вопрос «что это за продукт и какие решения уже приняты»;
- `Status.md` отвечает на вопрос «что уже сделано и какой следующий шаг».
