## Что за проект

Разрабатывается Telegram-бот для групповых fitness challenge в чатах.

Основная идея:

- администратор запускает challenge в групповом чате;
- бот ежедневно публикует задания с упражнениями;
- участники присоединяются к challenge;
- участники получают личные задания и напоминания;
- участники отмечают выполнение;
- бот ведет статистику по прогрессу.

## Цель MVP

Минимально полезная версия должна уметь:

- запускаться на Ubuntu-сервере в Docker;
- подключаться к PostgreSQL;
- подключаться к Telegram;
- хранить упражнения в БД;
- запускать challenge из группового чата;
- публиковать хотя бы один день challenge;
- позволять пользователю присоединиться;
- позволять пользователю отметить выполнение дня;
- сохранять историю выполнения;
- отправлять личные напоминания;
- переживать рестарт контейнеров без потери данных.

## Принятые технические решения

### Стиль кода

- Не нужно делать лишние переносы строк внутри методов.
- Если метод принимает `Object...`, при вызове не нужно создавать массив вроде `new String[]{...}`.
- Текстовые шаги диалогов должны выноситься из `UpdateHandler` в отдельные handler-классы.

### Структура пакетов

- В корневом пакете `su.msk.nlx2.challengebot` не должно быть пакета `bot`.
- Все сервисы должны лежать в пакете `service` и его подпакетах.
- Все модели данных должны лежать в пакете `model` и его подпакетах.
- Все enum должны лежать в пакете `model.type` и его подпакетах.
- Текущая договоренность по bot-слою:
  - верхнеуровневая обработка update лежит в `service.bot`;
  - диалоги и их состояние лежат в `service.bot.conversation`, `model.message` и `model.type.message`;
  - клавиатуры лежат в `service.bot.keyboard`;
  - общая инфраструктура handler-классов лежит в `service.bot.message`;
  - админские handler-классы лежат в `service.bot.message.admin`;
  - пользовательские handler-классы лежат в `service.bot.message.user`.

### Конфигурация и секреты

- Реальные секреты не хранятся в репозитории.
- Шаблон конфигурации хранится в `.env.example`.
- Реальные значения хранятся в локальном `.env`.
- На сервере также используется отдельный локальный `.env`.
- `.env` игнорируется через `.gitignore`.
- Редактируемые runtime-конфиги должны быть доступны на сервере вне контейнера через bind mount каталога `./config`.
- Каталог `./config` не должен храниться в Git и должен создаваться на сервере отдельной инициализацией из образа.
- На текущем этапе в `./config` после инициализации должны лежать:
  - `application.yml`
  - `logback-spring.xml`
  - `messages/messages_ru.properties`
  - `messages/messages_en.properties`

### Миграции базы данных

- Liquibase не должен запускаться автоматически при старте приложения.
- Миграции применяются отдельной явной командой.
- Разработчик должен видеть результат применения миграций в терминале.
- Источник миграций: `db/liquibase/changelog.xml`.
- Новые изменения схемы должны разбиваться на отдельные changeset.
- В одном changeset должен быть один узкий шаг изменения схемы.
- Не нужно объединять в один changeset несколько разных изменений таблиц, индексов и ограничений.

### Имена таблиц

- Таблицы должны называться в единственном числе.
- Таблица пользователя называется `tg_user`, а не `user`, потому что `user` является конфликтным именем для PostgreSQL.
- Таблица чатов называется `chat`.
- У `chat` нет отдельного surrogate `id`; primary key — `tg_chat_id`.
- Таблица `program` ссылается на чат через `chat_tg_id`.

### Локализация

- Локализация должна строиться поверх `MessageSource` и отдельных `messages_*.properties`.
- На текущем этапе используются `messages_ru.properties` и `messages_en.properties`.
- Если Telegram присылает неподдерживаемую локаль, приложение должно падать обратно на `ru`.
- В Docker бот сначала ищет локализации во внешнем каталоге `/app/external-config/messages`, затем во встроенных ресурсах jar.
- Тексты интерфейса и сообщений не должны оставаться захардкоженными в обработчиках.
- В таблице `tg_user` хранится поле `locale`.
- Порядок выбора языка для пользователя:
  - сначала `tg_user.locale`, если оно явно задано;
  - затем `message.from().languageCode()` из Telegram;
  - затем дефолт `ru`.
- Тексты кнопок должны локализоваться.
- Технические идентификаторы действий, callback data и внутренние ключи не должны зависеть от локализации.
- В Java-модели сохраненное значение из `tg_user.locale` называется `localeCode`.
- В начале обработки Telegram update resolver заполняет transient-поле `TgUser.locale` типа `Locale`; дальше обработчики используют его напрямую.

Текущее состояние реализации:

- уже добавлены `LocalizationConfig` и `LocalizationProperties`;
- уже добавлен сервис `BotMessages`;
- уже созданы `messages_ru.properties` и `messages_en.properties`;
- тексты `UpdateHandler`, `MessageSender` и фабрик клавиатур уже вынесены на ключи локализации;
- в коде должны оставаться только технические action/callback-константы и служебные алиасы команд;
- пользователь уже может менять язык через бота, и выбор сохраняется в `tg_user.locale`.
- отдельный `BotUserContext` не используется; роль, tg id и текущая локаль берутся из `TgUser`.

### Архитектура обработки сообщений

- `UpdateHandler` должен заниматься только routing и orchestration верхнего уровня.
- Разрешение пользователя и текущей локали должно выноситься в отдельный resolver-сервис.
- Обработка private message и callback query не должна разрастаться внутри `UpdateHandler`; её нужно выносить в отдельные service-классы.
- Текстовые шаги диалогов обрабатываются через `MessageHandlerProvider`.
- Для каждого `ConversationStep` должен быть отдельный `MessageHandler`.
- Если шаг диалога требует не текст, а другой тип Telegram message, например CSV-документ, `usersShared` или `chatShared`, это тоже должно обрабатываться через handler текущего `ConversationStep`.
- В `PrivateChatUpdateHandler` не нужно добавлять отдельные ветки под конкретные `ConversationStep`; при необходимости данные Telegram message нужно передать через `MessageHandlerContext`.
- Действия пользовательского меню в личке не должны зависеть от локализованного текста кнопок: использовать `InlineKeyboardMarkup` и техническое `callback_data`.
- Callback-действия пользовательского меню должны маршрутизироваться через отдельный provider + handler-классы (по аналогии с `MessageHandlerProvider`).
- Действия административного меню в личке также не должны зависеть от локализованного текста кнопок: использовать `InlineKeyboardMarkup` и техническое `callback_data`.
- Callback-действия административного меню должны маршрутизироваться через отдельный provider + handler-классы.
- Логика конкретных callback-действий меню должна жить в handler-классах; `PrivateChatUpdateHandler` не должен содержать их бизнес-реализацию.
- Пользователь и администратор должны идти по одному общему циклу обработки личных сообщений.
- Отличие администратора от обычного пользователя должно выражаться в дополнительных кнопках и ветках внутри общего цикла, а не в отдельном наборе session/service-классов.
- На текущем этапе используются:
  - `BotRequestContextResolver` для синхронизации пользователя и заполнения `TgUser.locale`
  - `PrivateChatUpdateHandler` для основного private-flow
  - `CallbackQueryUpdateHandler` для inline callback-ов
  - `ConversationService` + `ConversationSession` + `ConversationStep`
  - `MessageHandlerProvider` и набор handler-классов в `service.bot.message.admin` и `service.bot.message.user`
  - `UserMenuActionHandlerProvider` и набор callback-handler-классов для пользовательского inline-меню
  - `AdminMenuActionHandlerProvider` и набор callback-handler-классов для административного inline-меню

### Временное тестовое поведение

- Тестовая функциональность приветствия через кнопку `привет` удалена.
- Бот больше не должен отвечать тестовым hello-flow в группах и callback-ах.

### Деплой

- PostgreSQL и бот запускаются в Docker Compose.
- Миграции запускаются отдельным compose-сервисом `migrate`.
- Обновление приложения и обновление PostgreSQL считаются разными операциями.
- PostgreSQL не нужно перезапускать при каждом обновлении приложения.
- На текущем этапе контейнер `bot` по умолчанию публикует JDWP debug-порт `5005` для удаленной отладки.
- На текущем этапе этот порт открыт без ограничений и считается временным инфраструктурным решением.

### Логирование

- Логи приложения должны писаться в доступный путь.
- Используется `logback-spring.xml`.
- В Docker, если существует `config/logback-spring.xml`, используется внешний файл из `/app/external-config`.
- По умолчанию используется каталог `./logs` локально и bind-mounted каталог в Docker.
- Должно быть два файла логов:
  - `application-info.log` для логов уровня `INFO` и выше
  - `application-debug.log` для более подробных логов приложения
- Оба файла должны ротироваться.
- Debug и trace логи сторонних библиотек не должны попадать в файлы без явного включения.
- SQL-запросы Hibernate и bind-параметры должны попадать в `application-debug.log`.

### Telegram startup

- Если `BOT_TOKEN` пустой или равен `replace_me`, приложение может стартовать без polling.
- Это сделано для безопасной проверки приложения без реального токена.

### Ввод даты и времени в Telegram

- У Telegram Bot API нет штатного нативного date picker или time picker для обычного бота.
- На текущем этапе дата и время вводятся текстом.
- Timezone при создании challenge выбирается через готовые кнопки с распространенными `ZoneId`; если нужного варианта нет, администратор может ввести любой корректный `ZoneId` вручную.
- Если позже понадобится более удобный выбор, его можно сделать через:
  - собственную inline-клавиатуру-календарь;
  - Telegram Mini App.

## Продуктовые требования

### Роли

Есть три роли:

- администратор чата;
- участник challenge;
- система.

Техническое уточнение текущей реализации:

- на уровне Java-кода роль пользователя хранится как enum `UserRole`;
- в enum сейчас есть значения:
  - `USER`
  - `ADMIN`
- в таблице `tg_user` поле `role` хранится строкой;
- текущие строковые значения в БД:
  - `USER`
  - `ADMIN`

### Что делает администратор

Администратор должен иметь возможность:

- открыть административную панель в личном диалоге с ботом;
- добавить другого администратора;
- добавить бота в групповой чат;
- запустить challenge;
- задать параметры challenge;
- при необходимости остановить или завершить challenge;
- загружать упражнения;
- смотреть статистику.

Только администратор должен иметь доступ к административным командам.

### Как появляется первый администратор

На текущем этапе первый администратор заводится вручную в базе данных.

Текущее правило:

- в таблице `tg_user` должна существовать запись с нужным `tg_id`;
- у этой записи поле `role` в БД должно быть равно `ADMIN`;
- в коде это соответствует `UserRole.ADMIN`;
- после этого пользователь может открыть личный чат с ботом и получить административное меню.

Как practically получить запись пользователя в БД:

- любой пользователь должен написать боту любое сообщение или отправить `/start`;
- после этого бот должен создать или обновить запись пользователя в таблице `tg_user`.

Что происходит после ручной смены роли в БД:

- если пользователю в базе выставить `role = ADMIN`, удалять диалог с ботом не требуется;
- достаточно отправить боту следующее сообщение или снова написать `/start`;
- после этого бот должен показать административное меню.

Это временное решение для MVP-этапа. Позже можно добавить отдельный bootstrap-механизм.

### Административная панель

На текущем этапе административная панель работает в личном чате с ботом.

Если пользователь является администратором и отправляет `/start`, бот должен показать удобное меню.

На текущем этапе меню двухуровневое:

- в основном меню у администратора остаются обычные пользовательские действия;
- дополнительно есть кнопка `Администрирование`;
- внутри раздела `Администрирование` находятся все административные действия.

В разделе `Администрирование` должны быть доступны действия:

- `Добавить администратора`
- `Начать challenge`
- `Управление challenge`
- `Импорт CSV упражнений`

### Добавление администратора

Текущее решение:

- администратор выбирает пункт `Добавить администратора`;
- бот предлагает выбрать пользователя через Telegram `requestUsers`;
- после выбора пользователя бот создает или обновляет запись в `tg_user`;
- выбранному пользователю выставляется роль `UserRole.ADMIN`, которая сохраняется в БД как `ADMIN`.

Важно:

- сейчас используется не передача контакта, а встроенный механизм выбора пользователя через Telegram-кнопку;
- это удобнее и надежнее для текущей реализации.

### Создание challenge через личный диалог

Текущее решение для MVP:

- администратор выбирает пункт `Начать challenge`;
- бот предлагает выбрать чат через Telegram `requestChat`;
- после выбора чата бот пошагово запрашивает параметры.

На текущем этапе пошагово вводятся:

- длительность в днях;
- время публикации;
- timezone через готовые кнопки с возможностью ручного ввода;
- количество упражнений в день;
- количество типов упражнений в день;
- частота выходных `rest_day_frequency`.

Дата старта больше не вводится вручную:

- если администратор запускает challenge, датой старта автоматически становится завтрашний день;
- это сделано потому, что в обычном Telegram Bot API нет удобного встроенного выбора даты;
- если позже понадобится выбор даты, лучше сделать отдельную inline-клавиатуру-календарь или Telegram Mini App с календарем, где по умолчанию выбрана текущая дата.

После ввода параметров бот:

- показывает сводку;
- ждет подтверждение `Создать challenge`;
- сохраняет `Program` в БД;
- заранее создает `ProgramDay` на все дни challenge;
- отправляет подтверждение в выбранный чат;
- к стартовому сообщению challenge в выбранном чате прикрепляет кнопку `Участвовать`;
- возвращает администратора в главное меню.

### Управление существующими challenge

На текущем этапе администратор также может открыть список всех challenge, находящихся в статусах:

- `ACTIVE`
- `SCHEDULED`
- `PAUSED`

В этом списке администратор должен видеть основные параметры challenge и иметь возможность:

- поставить challenge на паузу;
- перевыложить последний опубликованный день;
- удалить challenge;
- подтвердить удаление отдельной кнопкой.

Текущее техническое поведение:

- пауза меняет `program.status` на `PAUSED`;
- перевыкладка последнего опубликованного дня удаляет старые сообщения общего поста в группе и публикует этот же день заново;
- при перевыкладке личные сообщения участникам заново не рассылаются, чтобы не спамить личку;
- удаление физически удаляет `Program`, связанные `ProgramDay`, `DayExercise` и `Completion`;
- удаление `Chat` при этом не выполняется.

### Импорт упражнений через CSV

На текущем этапе загрузка метаданных упражнений выполняется администратором через личный чат с ботом.

Текущее решение:

- администратор открывает раздел `Администрирование`;
- выбирает пункт `Импорт CSV упражнений`;
- бот переводит администратора в режим ожидания CSV-файла;
- администратор отправляет CSV как Telegram document;
- бот скачивает файл из Telegram, импортирует его и возвращает статистику:
  - сколько упражнений создано;
  - сколько упражнений обновлено.

Текущий формат CSV:

- `name`
- `type_name`
- `base_reps`
- `is_static_reps`
- `static_reps`
- `reps_unit`
- `video_path`
- `comment`

Текущие правила импорта:

- CSV должен содержать строку заголовков;
- `name` обязателен;
- `type_name` обязателен;
- `base_reps` должен быть целым числом 0 или больше;
- `is_static_reps` должен содержать логическое значение;
- `static_reps` должен быть целым числом 0 или больше;
- `reps_unit` может быть `repetitions` или `seconds`;
- если `reps_unit` отсутствует в CSV, используется значение по умолчанию `REPS`;
- `video_path` и `comment` могут быть пустыми;
- `type_name` может содержать один тип или несколько типов через `;`, например `спина;сгибатели рук`;
- если тип упражнения из `type_name` еще не существует, бот создает его автоматически;
- если упражнение с тем же `name` уже существует, его метаданные и набор типов обновляются;
- если такого упражнения еще нет, бот создает новое;
- поле `file_id` из Telegram этим импортом не заполняется и не перезаписывается.

Важно:

- на текущем этапе в CSV должен быть указан хотя бы один тип упражнения;
- связь упражнений и типов в БД many-to-many через таблицу `exercise_type_link`;
- пример CSV должен лежать в репозитории и использоваться как шаблон для ручного заполнения;
- CSV `examples/exercises-from-training-texts.csv` содержит первичную заготовку упражнений, собранную из старых текстовых challenge из локальной директории `/Users/argo/Yandex.Disk.localized/тренировки`; пути к видео в этой заготовке пока не заполнены, потому что при подготовке использовались только текстовые файлы.

### Отмена действия

Во время любого административного сценария должна быть доступна кнопка `Отмена`.

Поведение:

- текущий сценарий прерывается;
- состояние диалога очищается;
- администратор возвращается в основное меню.

### Что делает участник

Участник должен иметь возможность:

- вступить в challenge;
- указать максимальное количество подтягиваний;
- заполнить личные данные;
- получать ежедневные задания;
- получать личные напоминания;
- отметить выполнение дня;
- смотреть собственный прогресс.

### Личное меню пользователя

На текущем этапе в личном чате с ботом у пользователя есть отдельное личное меню.

Текущее поведение:

- при первом начале диалога бот должен запросить максимальное количество подтягиваний;
- пока пользователь не введет это значение, первичная настройка не считается завершенной;
- после сохранения значения бот показывает личное меню;
- если пользователь является администратором, в личном меню дополнительно показывается кнопка `Администрирование`, ведущая во вложенное административное меню.

В личном меню сейчас доступны действия:

- обновить максимальное количество подтягиваний;
- добавить напоминание;
- показать все свои напоминания;
- удалить напоминание кнопкой из списка;
- изменить язык интерфейса.

## Упражнения

Упражнение должно содержать:

- `id`
- название
- тип упражнения
- путь к видеофайлу или Telegram `file_id`
- базовое количество повторений для коэффициентной части
- коэффициент упражнения
- флаг, что упражнение имеет полностью статичное количество
- статичное количество повторений
- единицу измерения повторений `reps_unit`:
  - `REPS`
  - `SEC`
- комментарий

Важные правила:

- если видео есть, бот должен уметь отправить его;
- если видео нет, бот должен отправить текст;
- если `video_path` пустой или указывает на несуществующий файл, публикация не должна падать, вместо видео отправляется текст задания;
- упражнения должны храниться в БД;
- упражнение может быть связано с одним или несколькими типами упражнений;
- администратор уже может импортировать метаданные упражнений из CSV через личный чат;
- пример CSV лежит в `examples/exercises-metadata.csv`;
- единица измерения повторений хранится в `reps_unit`;
- если `reps_unit = REPS`, личное задание показывает количество в повторах;
- если `reps_unit = SEC`, личное задание показывает количество в секундах;
- упражнение может иметь фиксированную добавку `static_reps`, которая не зависит от коэффициентов;
- если `is_static_reps = true`, итоговое значение упражнения равно `static_reps`, а коэффициентная часть игнорируется;
- если `is_static_reps = false`, итоговое значение упражнения считается как сумма:
  - `static_reps`
  - значения, рассчитанного по коэффициентам.

## Типы упражнений

Система хранит справочник типов упражнений и связывает упражнения с типами many-to-many.

Стандартные типы добавляются миграцией Liquibase:

- спина
- пресс
- сгибатели ног
- разгибатели ног
- сгибатели рук
- разгибатели рук
- кардио
- Растяжка

Важно:

- на текущем этапе упражнение должно иметь хотя бы один тип при импорте из CSV;
- если упражнение относится к нескольким типам, в CSV они указываются через `;`, например `спина;сгибатели рук`.

## Challenge

Challenge запускается в конкретном групповом Telegram-чате.

При создании challenge администратор задает:

- чат;
- дату старта;
- длительность в днях;
- время ежедневной публикации;
- количество упражнений в день;
- количество типов упражнений в день;
- правило роста нагрузки по дням;
- частоту выходных дней `rest_day_frequency`.

Ожидаемый сценарий:

1. Администратор вызывает команду.
2. Бот проверяет, что это администратор.
3. Бот создает challenge.
4. Бот создает связанные дни challenge.
5. Бот подтверждает параметры запуска.

На текущем этапе это уже реализуется через личный пошаговый сценарий в административной панели.

Текущее уточнение:

- поле `program.rest_day_frequency` уже есть в схеме и модели;
- при создании challenge администратор вводит `rest_day_frequency`;
- значение `0` означает, что регулярные выходные дни не включены;
- значение `N > 0` означает один выходной после каждых `N` тренировочных дней, например `6` дает 6 дней тренировок и 1 день отдыха;
- день считается выходным, если `day_index % (rest_day_frequency + 1) == 0`.

## Ежедневная публикация дня

Каждый день бот должен публиковать новый день challenge в групповой чат.

В групповом сообщении должно быть:

- номер дня challenge;
- список упражнений;
- комментарии;
- видео, если доступны;
- кнопка `Участвовать`;
- без персонального количества повторений.

Почему в групповом сообщении нет повторений:

- количество может отличаться у разных пользователей;
- персональные значения должны отправляться пользователю в личку.

На текущем этапе реализован первый рабочий вариант daily workflow:

- scheduler раз в минуту проверяет `ACTIVE` и `SCHEDULED` challenge;
- для каждого challenge используется его `timezone` и `post_time`;
- если наступила дата и время публикации, бот берет `ProgramDay` на текущую дату;
- если текущий день является выходным по `rest_day_frequency`, бот публикует сообщение о выходном, не подбирает упражнения и отправляет участникам личное сообщение о выходном без повторений;
- если день еще `planned`, бот подбирает упражнения;
- выбранные упражнения сохраняются в `day_exercise`;
- бот отправляет общий пост в групповой чат;
- для каждого упражнения бот отправляет видео или текст;
- если путь к видео пустой или неверный, бот отправляет текстовое сообщение с тем же caption;
- `message_id` сообщений общего поста сохраняются в `program_day_message`, чтобы администратор мог перевыложить последний день;
- после публикации день получает статус `published`;
- если challenge был `SCHEDULED`, он переводится в `ACTIVE`;
- активным участникам этого challenge бот отправляет личную версию задания с количеством повторений.

После публикации:

- данные дня должны сохраняться в БД;
- повторная публикация того же дня не должна создавать дубль;
- уже опубликованный день не должен пересобираться задним числом.

Перевыкладка дня:

- администратор может нажать `Перевыложить последний день` для challenge;
- бот ищет последний `ProgramDay` со статусом `published`;
- бот пытается удалить все сохраненные сообщения общего поста этого дня из группового чата;
- удаление выполняется best-effort, потому что Telegram может отказать в удалении старых сообщений или сообщений без прав;
- после попытки удаления бот публикует тот же день заново;
- упражнения дня при этом не пересобираются, если они уже были сохранены в `day_exercise`.

## Подбор упражнений

Алгоритм подбора на MVP должен учитывать:

- `exercises_per_day`;
- лимит по числу типов упражнений;
- недопустимость повторов, если еще есть уникальные упражнения;
- нежелательность использовать типы упражнений, которые были в предыдущий день.

Для MVP достаточно простого алгоритма:

1. взять допустимые упражнения;
2. убрать неподходящие типы и упражнения;
3. выбрать нужное количество;
4. сохранить результат.

## Персональная нагрузка

Итоговая нагрузка рассчитывается по формуле:

`result = exercise_multiplier * day_multiplier * user_multiplier`

С учетом нового статичного компонента итоговое правило должно быть таким:

- сначала считается коэффициентная часть;
- затем к ней прибавляется `static_reps`;
- если у упражнения `is_static_reps = true`, то итоговое значение равно только `static_reps`.

Требования:

- расчет детерминированный;
- округление всегда вверх;
- рассчитанное значение сохраняется в БД;
- так как значения могут отличаться у разных пользователей, персональные цифры нельзя полагаться выводить только в общем сообщении.

Текущее MVP-правило персонального коэффициента:

- базовый пользователь считается как пользователь с `maxPullUps = 10`;
- коэффициент пользователя считается как `maxPullUps / 10`;
- минимальный коэффициент сейчас ограничен значением `0.2`, чтобы пользователь с `maxPullUps = 0` не получал полностью нулевую коэффициентную часть;
- это временное простое правило, которое позже можно заменить на более точную формулу.

Из этого следует архитектурное правило:

- в групповой чат идет общий пост дня;
- пользователю в личку отправляется персональная версия задания.

## Участие пользователя

После публикации стартового поста challenge пользователь должен иметь возможность вступить в challenge.

При вступлении:

1. создается или активируется запись участия;
2. пользователю могут быть показаны его личные характеристики для редактирования;
3. пользователь становится участником challenge;
4. ему становится доступна отметка выполнения;
5. ему отправляется личное сообщение с текущим заданием.

Нужно хранить:

- участие пользователя в конкретном challenge;
- дату вступления;
- активность участия;
- персональный коэффициент;
- настройки напоминаний;
- флаг возможности писать в личку.

Текущее состояние реализации:

- добавлена таблица `program_participant`;
- участие уникально по паре `program_id + user_id`;
- при нажатии кнопки `Участвовать` бот создает или активирует запись участия;
- на текущем этапе пользователь может одновременно участвовать только в одном challenge;
- если пользователь уже является активным участником любого текущего challenge, новое вступление блокируется;
- при заблокированном вступлении бот отправляет пользователю личное сообщение с ошибкой и показывает callback-ответ;
- участвовать можно только в challenge со статусом `ACTIVE` или `SCHEDULED`;
- если пользователь еще не указал `maxPullUps`, бот просит сначала открыть личный диалог и заполнить первичную настройку;
- если текущий день уже опубликован, после вступления бот отправляет пользователю личную версию задания за сегодня;
- если текущий день еще не опубликован, бот только сохраняет участие.
- если у пользователя есть активное участие, в личном меню показывается кнопка `Сдаюсь`;
- на текущем этапе кнопка `Сдаюсь` завершает все активные участия пользователя в текущих challenge через `program_participant.active = false`, не удаляя историю;
- позже, когда появится просмотр своих active challenge, `Сдаюсь` нужно будет сделать выборочным для конкретного challenge.
- `BotRequestContextResolver` заполняет transient-флаг `TgUser.activeParticipant`, чтобы меню могло показывать кнопку `Сдаюсь` без отдельного DTO состояния.

## Напоминания

Пользователь должен иметь возможность настроить личные напоминания.

Правила:

- время задает сам пользователь;
- напоминание отправляется активным участникам;
- если текущий день выполнен, напоминание не отправляется;
- если пользователь отстает, можно напоминать о последнем невыполненном дне;
- если бот не может написать первым в личку, это должно корректно обрабатываться.

## Отметка выполнения

Участник должен иметь возможность отметить выполнение дня.

Правила:

- отметка привязана к конкретному `program_day`;
- один пользователь может отметить день не более одного раза;
- время отметки сохраняется;
- должно быть различимо:
  - не участвует;
  - участвует, но не выполнил;
  - выполнил.

Отмена отметки допустима как будущее улучшение, но не обязательна для MVP.

## Надежность и устойчивость

Система должна быть устойчивой к повторным действиям и рестартам.

Ключевые требования:

- повторный запуск daily job не создает дублей;
- повторное нажатие кнопки участия безопасно;
- повторная отметка выполнения безопасно обрабатывается;
- история опубликованных дней не меняется;
- после рестарта контейнера работа продолжается за счет БД.

## Критические ограничения текущей кодовой базы

- Вместо `participant_reminders` на текущем этапе уже добавлена таблица `user_reminder`.
- Схема `exercise` уже использует типы упражнений через `exercise_type` и `exercise_type_link`, но отдельного типа коэффициента кроме `reps_unit` пока нет.
- Персональный snapshot рассчитанной нагрузки пока не сохраняется отдельно на каждого участника.
- Обработка ошибок Telegram-отправки в daily workflow пока минимальная.
- первый администратор пока добавляется вручную в БД.
- Проверка реального Docker startup из текущего агентского окружения пока не выполнена, потому что здесь нет доступного Docker daemon.

## Как продолжать работу после потери контекста

1. Сначала прочитать `Status.md`.
2. Посмотреть, какой этап активен сейчас.
3. Проверить `README.md`, чтобы не сломать текущий deployment flow.
4. Перед изменением поведения обновлять этот файл.
5. Перед закрытием этапа обновлять `Status.md`.

Главное правило:

- `Promt.md` отвечает на вопрос «что это за продукт и какие решения уже приняты»;
- `Status.md` отвечает на вопрос «что уже сделано и какой следующий шаг».
