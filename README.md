# Challenge Bot

## Текущее состояние

- Есть Spring Boot приложение с базовой интеграцией Telegram Bot API.
- Есть PostgreSQL через Spring Data JPA.
- Миграции Liquibase запускаются отдельно, а не при старте приложения.
- Есть Dockerfile и Docker Compose для бота и базы данных.

Бизнес-логика challenge пока реализована не полностью. На текущем этапе задача репозитория — дать стабильную основу для деплоя и дальнейшей разработки.

## Требования

- Java 25
- Maven Wrapper `./mvnw` с Maven 3.9.14
- Docker и Docker Compose

Для локальной сборки используй Maven Wrapper, чтобы не зависеть от версии Maven, установленной в системе или встроенной в IntelliJ:

```bash
./mvnw clean test
```

Причина: Maven 3.9.11 из bundled Maven в IntelliJ на JDK 25 печатает warning из Guice про `sun.misc.Unsafe`.
Maven 3.9.14 под JDK 25 этот warning не печатает.

## Конфигурация

Создай `.env` на основе `.env.example` и заполни его реальными значениями.

Важно:

- `.env` должен храниться только локально на машине разработчика и на сервере, в репозиторий его коммитить нельзя;
- `.env.example` хранится в репозитории как шаблон без секретов;
- `DATABASE_URL` в `.env` используется для локального запуска через `mvn spring-boot:run`.
- В Docker Compose для контейнера бота он переопределяется на `jdbc:postgresql://postgres:5432/challenge`.
- Если `BOT_TOKEN` пустой или равен `replace_me`, приложение стартует, но Telegram polling не запускается.

Практический вариант хранения секретов для этого проекта:

- реальные значения хранить в локальном `.env`, который игнорируется Git;
- на сервере также хранить отдельный локальный `.env`;
- для резервного хранения пароля БД лучше использовать менеджер паролей или зашифрованную заметку, а не репозиторий.

По ролям пользователей:

- в коде роль пользователя хранится как enum `UserRole`;
- в базе данных поле `tg_user.role` хранится строкой;
- используется стандартный JPA/Hibernate mapping через `@Enumerated(EnumType.STRING)`;
- допустимые значения в БД на текущем этапе: `USER` и `ADMIN`.

По дате и времени в Telegram:

- у Telegram Bot API нет штатного встроенного выбора даты и времени для обычного бота;
- поэтому сейчас дата и время вводятся текстом;
- позже это можно улучшить собственной inline-клавиатурой или Mini App.

По пользователям:

- в `tg_user` добавлено поле `locale` для сохраненного языка пользователя;
- в Java-модели сохраненное значение называется `TgUser.localeCode`, а текущая разрешенная локаль запроса хранится в transient-поле `TgUser.locale`;
- в `tg_user` хранится `max_pull_ups` как первое персональное значение для оценки формы;
- личные времена напоминаний хранятся в отдельной таблице `user_reminder`;
- это сделано отдельной таблицей, а не JSON-полем, чтобы проще было валидировать, обновлять и использовать в scheduler.
- при первом личном диалоге бот запрашивает у пользователя `max_pull_ups`;
- главное пользовательское меню в личке сделано через `InlineKeyboardMarkup`, а действия идут через техническое `callback_data`;
- `PrivateChatUpdateHandler` больше не парсит локализованный текст пользовательских кнопок;
- после этого в личке доступны действия:
  - изменить максимум подтягиваний;
  - добавить напоминание;
  - показать свои напоминания;
  - удалить напоминание кнопкой из списка.

По административному меню:

- у администратора в главном меню есть кнопка `Администрирование`;
- само админ-меню сделано через `InlineKeyboardMarkup` + `callback_data`;
- `PrivateChatUpdateHandler` больше не парсит локализованный текст админ-кнопок;
- внутри этого раздела доступны:
  - добавление администратора;
  - создание challenge;
  - управление существующими challenge;
  - импорт метаданных упражнений из CSV.

По схеме БД:

- таблицы называются в единственном числе;
- таблица пользователя называется `tg_user`, потому что `user` конфликтует с PostgreSQL;
- Java-сущность пользователя называется `TgUser`, чтобы не конфликтовать с Telegram-классом `com.pengrad.telegrambot.model.User`;
- таблица чатов называется `chat`;
- у `chat` нет отдельного `id`, primary key — `tg_chat_id`;
- `program.chat_tg_id` ссылается на `chat.tg_chat_id`;
- в `program` есть поле `rest_day_frequency`: `0` означает отсутствие регулярных выходных, `N > 0` означает один выходной после каждых `N` тренировочных дней.

По импорту упражнений из CSV:

- импорт запускается администратором через личный чат с ботом;
- файл нужно отправлять как Telegram document;
- шаблон CSV лежит в `examples/exercises-metadata.csv`;
- заготовка CSV, собранная из старых текстовых challenge, лежит в `examples/exercises-from-training-texts.csv`;
- текущие колонки CSV:
  - `name`
  - `type_name`
  - `base_reps`
  - `is_static_reps`
  - `static_reps`
  - `reps_unit`
  - `video_path`
  - `comment`
- `reps_unit` может быть `repetitions` или `seconds`;
- если `reps_unit` не указан, импорт использует `repetitions`;
- `type_name` обязателен и может содержать один или несколько типов упражнений через `;`, например `спина;сгибатели рук`;
- стандартные типы упражнений, которые добавляются Liquibase: `спина`, `пресс`, `сгибатели ног`, `разгибатели ног`, `сгибатели рук`, `разгибатели рук`, `кардио`, `Растяжка`;
- если тип из `type_name` не существует, он будет создан автоматически;
- если упражнение с тем же `name` уже существует, запись будет обновлена, включая набор типов;
- если упражнения еще нет, запись будет создана;
- связь упражнений и типов хранится как many-to-many через таблицу `exercise_type_link`.

По ежедневной публикации challenge:

- бот раз в минуту проверяет `ACTIVE` и `SCHEDULED` challenge;
- для публикации используются `program.timezone` и `program.post_time`;
- при создании challenge timezone выбирается кнопкой из готовых `ZoneId`, но можно ввести значение вручную;
- при создании challenge администратор задает `rest_day_frequency`;
- дата старта при создании challenge ставится автоматически на завтрашний день;
- к стартовому сообщению challenge сразу прикрепляется кнопка `Участвовать`;
- общий пост в групповой чат отправляется без количества повторений;
- в выходной день бот публикует сообщение о выходном и не подбирает упражнения;
- в общем посте есть кнопка `Участвовать`;
- если `video_path` пустой или неверный, бот отправит текстовое задание вместо видео;
- пользователь должен заранее открыть личный чат с ботом и указать `max_pull_ups`;
- после участия пользователь получает личное задание с количеством повторений;
- повторное нажатие `Участвовать` активным участником не отправляет личное задание повторно, а отправляет ошибку вступления;
- на текущем этапе пользователь может участвовать только в одном challenge одновременно;
- если пользователь уже участвует в challenge, новая попытка вступления блокируется, а ошибка отправляется ему в личный чат;
- если пользователь участвует в challenge, в личном меню появляется кнопка `Сдаюсь`, которая на текущем этапе завершает все активные участия пользователя без удаления истории.
- активные участники получают личное задание автоматически после публикации дня.

По перевыкладке дня:

- бот сохраняет `message_id` сообщений общего поста в таблицу `program_day_message`;
- администратор может нажать `Перевыложить последний день` в управлении challenge;
- бот best-effort удалит старые сообщения общего поста и опубликует этот же день заново;
- личные задания участникам при ручной перевыкладке заново не рассылаются.

По Liquibase:

- новые изменения схемы нужно разбивать на отдельные changeset;
- не нужно объединять в один changeset несколько разных изменений таблиц, индексов и ограничений.

По локализации:

- язык пользователя будет храниться в `tg_user.locale`;
- порядок выбора языка планируется такой:
  - сначала `tg_user.locale`;
  - затем `message.from().languageCode()` из Telegram;
  - затем дефолт `ru`;
- тексты будут выноситься в `messages_*.properties`;
- уже добавлены:
  - `LocalizationConfig`
  - `LocalizationProperties`
  - `BotMessages`
  - `messages_ru.properties`
  - `messages_en.properties`
- тексты в `UpdateHandler`, `MessageSender`, `AdminKeyboardFactory` и `UserKeyboardFactory` уже переведены на ключи локализации;
- пользователь уже может менять язык через бота, и выбор сохраняется в `tg_user.locale`;
- отдельный `BotUserContext` не используется: обработчики получают `TgUser` с уже заполненной runtime-локалью.
- файлы локализации сгруппированы по разделам с комментариями, чтобы ими было удобно пользоваться и дополнять.
- для Docker редактируемые runtime-конфиги не хранятся в Git;
- они один раз выгружаются из образа в локальный каталог `config/` командой:

```bash
./scripts/init-config.sh
```

- после этого можно редактировать:
  - `config/application.yml`
  - `config/logback-spring.xml`
  - `config/messages/messages_ru.properties`
  - `config/messages/messages_en.properties`

По логам:

- используется `logback-spring.xml`;
- `LOG_DIR` задает каталог для логов;
- создаются два файла:
  - `application-info.log`
  - `application-debug.log`
- оба файла ротируются автоматически;
- debug-логи сторонних библиотек в файлы не пишутся, если не включать их явно;
- SQL Hibernate и bind-параметры пишутся в `application-debug.log`, чтобы можно было контролировать реальные запросы в БД.

По debug-подключению:

- контейнер `bot` по умолчанию стартует с открытым JDWP-портом `5005`;
- к нему можно подключаться удаленным дебаггером;
- сейчас порт открыт без дополнительных ограничений;
- позже это нужно будет ограничить firewall, SSH tunnel или VPN.

## Локальный запуск

1. Поднять PostgreSQL:

```bash
docker compose up -d postgres
```

2. Применить миграции:

```bash
docker compose --profile tools run --rm migrate
```

3. Запустить приложение:

```bash
mvn spring-boot:run
```

Логи по умолчанию пишутся в каталог `./logs`.

## Запуск в Docker

1. Поднять PostgreSQL:

```bash
docker compose up -d postgres
```

2. Применить миграции:

```bash
docker compose --profile tools run --rm migrate
```

3. Выгрузить редактируемые runtime-конфиги:

```bash
./scripts/init-config.sh
```

4. Поднять бота:

```bash
docker compose up -d --build bot
```

После этого удаленный debug доступен на порту `5005`.
Если менялись файлы в `config/`, достаточно перезапустить контейнер `bot`.

## Деплой на Ubuntu сервер

Минимальный сценарий:

1. Скопировать проект на сервер.
2. Создать `.env` из `.env.example`.
3. Заполнить реальный `BOT_TOKEN`.
4. Поднять PostgreSQL:

```bash
docker compose up -d postgres
```

5. Применить миграции:

```bash
docker compose --profile tools run --rm migrate
```

6. Выгрузить редактируемые runtime-конфиги:

```bash
./scripts/init-config.sh
```

7. При необходимости отредактировать файлы в `config/`.

8. Поднять бота:

```bash
docker compose up -d --build bot
```

После запуска бота можно подключаться удаленным дебагом к `SERVER_IP:5005`.
Все редактируемые runtime-конфиги доступны на сервере в каталоге `config/`.

## Обновление на новую версию

Если меняется только код приложения или миграции, а PostgreSQL уже работает, достаточно:

```bash
git pull
docker compose --profile tools run --rm migrate
docker compose up -d --build bot
```

Почему здесь нет `docker compose up -d postgres`:

- если PostgreSQL уже поднят и его конфигурацию менять не нужно, повторно трогать его не надо;
- если позже ты поменяешь образ PostgreSQL в `docker-compose.yml`, то отдельное обновление базы стоит делать осознанно и вручную, потому что смена версии PostgreSQL поверх старого volume может требовать отдельной процедуры миграции данных.

Данные базы хранятся в Docker volume `pgdata`.

## Удаленный debug

Контейнер бота по умолчанию публикует JDWP-порт `5005`.

Подключение из IntelliJ IDEA:

1. Создать конфигурацию `Remote JVM Debug`.
2. Указать host сервера.
3. Указать port `5005`.

Если не хочется открывать debug наружу напрямую, можно подключаться через SSH tunnel:

```bash
ssh -L 5005:localhost:5005 user@SERVER_IP
```

Тогда в IDE нужно подключаться к `localhost:5005`.

## Миграции базы данных

- Changelog: `db/liquibase/changelog.xml`
- Миграции применяются вручную отдельной командой:

```bash
docker compose --profile tools run --rm migrate
```

- Предпросмотр SQL без применения:

```bash
docker compose --profile tools run --rm migrate updateSQL
```

- История примененных migration:

```bash
docker compose --profile tools run --rm migrate history
```

По умолчанию команда `docker compose --profile tools run --rm migrate` выводит логи прямо в терминал. Так как используется `--rm`, контейнер удаляется после завершения, поэтому постоянного места для просмотра этих логов после выполнения нет. Если нужно сохранить вывод, его лучше сразу перенаправить в файл или запускать без `--rm`.

## Текущие ограничения

- персональный snapshot нагрузки по каждому участнику пока не сохраняется отдельной таблицей;
- обработка ошибок Telegram-отправки в daily workflow пока минимальная;
- полноценная статистика challenge пока не реализована.
